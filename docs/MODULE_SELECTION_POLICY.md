# Module Selection Policy

Major modules must not begin because they are interesting or easy to implement. They must begin because ChatGPT-led review finds that they serve the final profitability goal and fit the upstream/downstream project sequence.

## Selection Requirements

Before implementation, each module requires:

- reason to do it now
- profitability relevance
- upstream dependencies
- downstream consumers
- interface boundaries
- conflict check with existing modules
- open-source alternatives review
- adopt, wrap, borrow, prototype, defer, or avoid recommendation
- implementation boundary
- success criteria
- prohibited scope
- next-module impact

## Codex Boundary

Codex documents and executes the approved scope. Codex does not independently select major modules or choose open-source frameworks.
